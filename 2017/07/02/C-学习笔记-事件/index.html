事件的由来

使用委托实现不同的打招呼方式

将多个方法绑定到同一个委托变量，则可以这样写

面向对象的一大特性就是封装，于是我们想到应该把 GreetingManager 类中 GreetPeople 方法所需要的委托变量也封装在该类中。

但是这里的问题是，我们应该将 Delegate1 的访问修饰符写成 public 还是 private。
如果写成 public ，那么在类之外就可以对它进行随意的赋值等操作，并不符合对象的封装性。
如果写成 private ，那么外界就无法访问到它，那么它也就失去了它的意义。
还有，如果它是 public ，外界对它进行注册时，第一次是赋值，用 = ，因为要实例化 ( 初始化？ ) ，第二次使用 +=，是注册。但是不管是赋值还是注册，都是将方法绑定到委托变量上，除了调用顺序不同外，没有什么差别。

综上所述，再联想到我们通常把字段修饰为 private，然后为了能让外界访问，又通过 public 的属性对其进行封装 ( 保护 ) ，那么我们也应该将委托变量定为 private，然后再想办法对其进行封装 ( 保护 ) 。

事件(Event)的作用就是封装委托变量。

范例说明

编写一个类模拟热水器，当水温达到 95 度时，发出警告，同时显示水温。

Observer 设计模式简介

上面的代码可以完成需求，但是进一步想，热水器应当仅用于加热水，而提示水温应当由显示器去做，发出警告应当由警报器去做，三个设备各司其职。

但是当水温高于 95 度时，应如可通知警告器和显示器呢？

Observer 设计模式主要包含两类对象：监视者 ( Observer ) 和监视对象 ( Subject ) 。

监视对象包含其他对象感兴趣的的内容，这里就是温度，而监视对象就是热水器。当水温高于 95 度时，会不断把数据发给监视它的对象。

监视者监视监视对象，当监视对象中的某件事发生时，会告知监视者，从而使监视者采取某些行动。这里警告器和显示器就是监视者。

本例中，事情的发生顺序如下描述

  1. 警报器和显示器告诉热水器，它对它的温度比较感兴趣(注册)。
  2. 热水器知道后保留对警报器和显示器的引用。
  3. 热水器进行烧水这一动作，当水温超过95度时，通过对警报器和显示器的引用，自动调用警报器的MakeAlert()方法、显示器的ShowMsg()方法。

Observer 设计模式是为了定义对象间的一种一对多的依赖关系，以便于当一个对象的状态改变时，其他依赖于它的对象会被自动告知并更新。Observer模式是一种松耦合的设计模式。
